## Context

The Foundry now owns the canonical fleet catalog, operations tooling, PSI Swarm, API, and private Cockpit. Performance evidence is fragmented:

- PSI Swarm produces useful repeated Lighthouse distributions and local SQLite history for web pages.
- The operations scripts write dated LCP scoreboards, but Cockpit does not consume a durable normalized receipt.
- Browser Resource Timing emits aggregated `api_call_timing` PostHog events, while the current Cockpit latency query reads `foundry_trace`; empty provider credentials silently become an empty dashboard.
- Backend timing coverage and event naming vary across products.
- The existing observability inventory proves source configuration, not live delivery or performance.

The fleet is hub-and-spoke: products may push evidence to Foundry, but they must not call one another. The designated operations host will run scheduled synthetic checks after a separately approved host cutover. The owner is the only current Cockpit user, so the interface can expose dense operational detail, but it must still avoid collecting sensitive payloads.

This design records proposed defaults. Implementation MUST NOT begin until the owner approves the decision table in Open Questions.

## Goals / Non-Goals

**Goals:**

- Answer web speed, API speed, traffic, error, regression, coverage, and freshness questions from one private Cockpit surface.
- Preserve source provenance and distinguish synthetic lab data, browser RUM, server spans, and provider enrichments.
- Show recent sanitized API activity, top and slow routes, and downstream operation/query timing.
- Continue working when PostHog or Cloudflare analytics are unavailable.
- Reuse PSI Swarm and the canonical catalog rather than create parallel registries or runners.
- Make initial rollout observational and reversible.

**Non-Goals:**

- Full request replay, distributed payload tracing, session replay, or raw SQL inspection.
- Capturing request/response bodies, URL query values, SQL bind values, headers, cookies, tokens, IP addresses, or user content.
- Replacing PostHog product analytics, Cloudflare platform metrics, or PSI Swarm's local diagnostic UI.
- Automatically blocking deploys or changing production rate limits.
- Probing mutating or authenticated business endpoints during the first rollout.

## Decisions

### 1. Foundry receipts are canonical; providers are enrichments

Foundry D1 stores a small provider-neutral evidence model. Synthetic runners and runtime adapters submit versioned receipts through authenticated `/v1/performance/*` endpoints. PostHog, Cloudflare, CrUX, and future OpenTelemetry adapters may enrich the same model, but absence of any provider cannot erase synthetic coverage.

This avoids coupling the control plane to PostHog pricing, retention, event naming, or credentials. Querying providers directly remains useful for diagnostics but is not a fleet contract.

Alternatives considered:

- PostHog as the sole store: simplest now, but current event drift already demonstrates the coupling risk.
- Cloudflare Analytics Engine as the sole store: attractive for Workers, but excludes non-Cloudflare and local/desktop surfaces.
- Local SQLite only: excellent for PSI Swarm detail, but unavailable to the private Cockpit unless the operations host is online and directly reachable.

### 2. Keep short sanitized spans and longer aggregate receipts

Two records serve different questions:

- `performance_spans`: sampled recent API/server/downstream operations for the “latest” view. Proposed retention is 7 days.
- `performance_rollups`: immutable percentile/count/error aggregates for trends. Proposed retention is 13 months, allowing year-over-year inspection without retaining user-level activity.

Every record includes project, surface, environment, revision when known, source, observation window, sample count, and ingestion timestamp. Percentiles are p50, p75, p95, and p99; web receipts may additionally retain PSI Swarm's p90 for compatibility.

Span fields are restricted to method, normalized route template, status class, duration, TTFB when known, cold/warm classification, trace correlation ID generated by the runtime, and sanitized downstream operation metadata. Cardinality limits reject unbounded route names.

### 3. Fingerprints, never raw queries

Downstream query visibility records:

- system kind (`d1`, `sql`, `kv`, `r2`, `external-http`, `ai`, `queue`, or `other`),
- a stable fingerprint hash,
- an allowlisted operation label such as `projects.list` or normalized HTTP host + route template,
- duration, success/failure, parent trace ID, and timestamp.

Literal SQL, bind values, URL query strings, payloads, stack traces containing values, and credentials are rejected by contract and by server-side validation. Runtime adapters normalize before transmission; the ingestion API validates again.

This gives “which operation is slow?” without turning Foundry into a sensitive data warehouse.

### 4. Catalog declares surfaces and safe probes

`catalog/foundry.json` remains the only hand-edited registry. Each maintained product may declare zero or more performance surfaces:

- public web URLs for PSI Swarm,
- public anonymous API health/read-only probe endpoints,
- expected method/status/timeout,
- criticality and optional owner-approved budgets,
- runtime evidence adapters expected for that surface.

Generated operations and Cockpit views derive from the catalog. A missing declaration appears as `unmeasured`; it is never silently treated as healthy.

### 5. Synthetic cadence is asymmetric

Proposed operations-host schedule:

- API: lightweight daily runs, five cold and fifteen warm samples per declared endpoint, bounded concurrency and jitter.
- Web: weekly PSI Swarm runs, five desktop and five mobile samples per critical URL.
- Release: an operator-triggered scoped run for changed surfaces.

Cold runs use fresh connections where the client supports it; warm runs reuse the client session. DNS, connect, TLS, TTFB, and total timing are captured where the runtime exposes them. The receipt marks unavailable phases instead of inventing zeroes.

### 6. Runtime instrumentation is sampled and non-blocking

Maintained APIs receive a small provider-neutral timing adapter using platform primitives already available in each stack. It emits bounded spans asynchronously and never delays or fails the product request when telemetry delivery fails. The default proposed sample is 10% of successful requests and 100% of errors or requests slower than the owner-approved slow threshold, capped per route and minute.

Existing `api_call_timing` and `foundry_trace` events get compatibility adapters during migration. New code uses one versioned Foundry naming contract. Browser RUM remains a separate source because it measures user-perceived network latency that server spans cannot.

### 7. Cockpit uses one dense Speed workspace

Add `/fleet/speed` under Visibility with:

- coverage/freshness cards and explicit `fresh`, `stale`, `unmeasured`, and `failing` states;
- fleet table with web p75/p95, API p50/p95/p99, request count, error rate, last sample, source, and revision;
- top API calls by volume, slowest route templates, highest error routes, and recent sanitized spans;
- route detail with latency and throughput trends, cold/warm split, status classes, and downstream operation waterfall summaries;
- web detail linking to PSI Swarm diagnostics and showing Core Web Vitals distributions;
- regressions with absolute and percentage deltas, evidence windows, and sample counts;
- source filters for synthetic, RUM, runtime, PostHog, Cloudflare, and imported evidence.

The existing static `/fleet/observability` inventory remains the configuration/integrity view and links to Speed. The current PostHog-only latency card is retired after parity.

### 8. Observe before enforcing budgets

For the first 14 days, the system reports regressions but does not page or block deployment. Suggested budgets may be calculated from baseline distributions, but only owner-approved per-surface budgets become active. Even after approval, the first enforcement mode is digest/alert; deploy blocking is a separate later decision.

A regression requires both a relative threshold and a minimum absolute delta plus minimum sample count, preventing tiny fast-route changes from creating noise.

## Risks / Trade-offs

- [Synthetic measurements do not equal real user experience] -> Label evidence sources prominently and show RUM/CrUX beside synthetic data when available.
- [Runtime telemetry adds cost and latency] -> Sample, batch, cap cardinality, deliver asynchronously, and make failure non-blocking.
- [Route or query labels leak data] -> Normalize at source, validate at ingestion, reject query strings and high-cardinality labels, and test the redaction contract.
- [A single operations-host region biases latency] -> Record probe region/host and support additional probe origins later without changing the receipt schema.
- [Cold/warm classification is imperfect on serverless platforms] -> Treat it as probe-client connection state unless runtime evidence explicitly reports platform cold start.
- [D1 retention grows unexpectedly] -> Keep spans short-lived, store aggregate rollups, expose ingestion volume, and run bounded cleanup as an explicit scheduled job.
- [Provider data disagrees] -> Never merge unlike sources into one percentile; display provenance and comparable windows.
- [Fleet-wide rollout becomes a broad rewrite] -> Ship the contract, Foundry ingestion, and synthetic coverage first; instrument maintained APIs incrementally and report missing runtime coverage.

## Migration Plan

1. Add contracts, catalog schema, validation, D1 migration, authenticated ingestion/query API, and tests without enabling any schedule.
2. Adapt existing PSI Swarm scoreboards into performance receipts and run a local-only fleet web/API smoke against a small allowlist.
3. Build Cockpit Speed against fixture/local D1 evidence and retire no existing view.
4. Add compatibility importers for current `api_call_timing` and `foundry_trace` sources.
5. Instrument a canary API (SaaS Maker) and verify redaction, cardinality, telemetry overhead, and failure isolation.
6. Roll out maintained API adapters in bounded project PRs; unknown coverage remains visible.
7. After explicit approval, install the daily/weekly schedules on the designated operations host.
8. Observe for 14 days, review baselines with the owner, and separately approve any alert budgets.

Rollback is additive: disable schedules and adapters, retain existing provider telemetry, hide the Speed navigation entry if necessary, and leave receipts readable. The migration introduces no required dependency for product request handling.

## Open Questions

Owner approval is required for these proposed defaults before implementation:

1. Canonical store: Foundry D1 receipts, with PostHog/Cloudflare as optional enrichments.
2. Probe scope: anonymous health and explicitly declared read-only endpoints only; no authenticated business probes initially.
3. Cadence: daily API, weekly web, and operator-triggered release checks.
4. Retention: 7 days for sampled sanitized spans and 13 months for aggregates.
5. Sampling: 10% successes, 100% errors/slow requests, with per-route caps.
6. Enforcement: 14 observation-only days, then alerts only after owner-approved budgets; no deploy blocking.
7. Query detail: fingerprints and allowlisted operation labels only, never raw query text or values.
